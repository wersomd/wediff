# Долги: рассрочка + «не влияет на баланс»

## Проблема

1. Каждый долг обязательно создаёт транзакцию по счёту (`disbursementTransactionId`),
   поэтому добавление долга раздувает баланс. Но старые долги (деньги получены давно,
   в текущем балансе их нет) не должны влиять на баланс.
2. Халяльная рассрочка без процентов — получена **вещь**, а не деньги на счёт.
   Нужно вести как долг с помесячным графиком, но без прихода на баланс.

## Решение

Общая идея: долг может существовать **без проводки по счёту**. Плюс новый тип долга
«рассрочка» с автоматическим графиком помесячных платежей.

### Модель данных

- `enum DebtKind { SIMPLE, INSTALLMENT }`, поле `Debt.kind` (дефолт `SIMPLE`).
- `disbursementTransactionId` уже nullable — используем для «без баланса».
- Новая таблица `DebtInstallment`:
  - `id`, `debtId`, `seq` (1..N), `dueDate`, `amount Decimal(14,2)`,
    `status: PENDING | PAID` (`enum InstallmentStatus`), `paidOn?`,
    `paymentId?` (ссылка на `DebtPayment`, чтобы `paid`/`remaining` считались
    существующей логикой суммирования платежей).
  - `@@index([debtId])`, `@@unique([debtId, seq])`.

### Создание долга

- **Обычный долг** (`SIMPLE`): чекбокс «Провести через счёт» (по умолчанию вкл).
  - Вкл → как сейчас: создаётся доход (я должен) / расход (мне должны) на счёт.
  - Выкл → транзакция не создаётся, `accountId` необязателен. Закрывает старые долги.
- **Рассрочка** (`INSTALLMENT`): только направление «я должен».
  - Поля: полная сумма, срок в месяцах `N`, дата первого платежа.
  - Проводка по счёту всегда выключена (получена вещь, а не деньги).
  - График генерится при создании: `per = round(total / N, 2)` на каждый месяц,
    последний месяц = `total − per*(N−1)` (поглощает округление). Даты: первый —
    указанная дата, далее +1 месяц.

### Погашения

- **SIMPLE**: как сейчас (`addPayment` / `deletePayment`), реальный расход/приход
  по счёту.
- **INSTALLMENT**: действия `payInstallment(installmentId, accountId, paidOn)` и
  `unpayInstallment(installmentId)`.
  - `payInstallment`: создаёт расход по счёту на сумму месяца, создаёт `DebtPayment`
    (для единого подсчёта `paid`), помечает месяц `PAID` + `paidOn` + `paymentId`.
    Когда все месяцы `PAID` → долг `PAID`.
  - `unpayInstallment`: удаляет транзакцию и `DebtPayment`, месяц → `PENDING`,
    долг → `OPEN`.

### Итоги и просрочка

- `remaining` = `principal − sum(payments)` (без изменений формулы). Для рассрочки
  = сумма ещё не оплаченных месяцев.
- `computeDebtTotals` учитывает рассрочку как обычный `I_OWE`.
- Долги без проводки учитываются в остатках/итогах, но на баланс счетов не влияют
  (транзакции просто нет).
- Просрочка рассрочки — по `dueDate` ближайшего `PENDING` месяца (переиспользуем
  логику `isOverdue`).

### UI

- Форма долга: селект типа (Обычный / Рассрочка), чекбокс «Провести через счёт»
  (для обычного), поля срока и даты первого платежа (для рассрочки).
- В списке: у рассрочки бейдж «Рассрочка», строка «следующий платёж k/N — дата»,
  раскрываемый график месяцев с кнопкой «Оплатить» на ближайшем неоплаченном.

## Вне объёма (YAGNI)

- Проценты / переплата (рассрочка халяльная, 0%).
- Push-уведомления. «Напоминание» = подсветка ближайшего/просроченного месяца в UI.
- Неравные платежи / первоначальный взнос (пока равные доли).
